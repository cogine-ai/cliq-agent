# v0.8 Runner Integration — Design Spec

**Status:** approved 2026-05-07 (brainstorming session)
**Branch:** `cliq/v0.8-runner-integration`
**Parent:** `cliq/v0.7-tx-runtime-spec` (53 commits implementing Phases 1–14 of the v0.8 plan)
**Reference:** OpenAI Codex CLI `SessionTask` / `ToolOrchestrator` patterns (see Architecture §A.1).

## Goal

Wire the transactional workspace runtime (already implemented as a standalone module) into the runner's turn lifecycle so `cliq run` and `cliq "..."` actually use it. Today the coordinator works only against manually-constructed approved transactions. After this change:

- A user running `cliq --tx edit "fix the foo bug"` gets an implicit per-turn tx that opens at turn start, accumulates edits in an overlay, validates at turn end, and applies if `applyPolicy` permits.
- `cliq tx open foo` followed by multiple turns accumulates edits across turns into one explicit tx, with apply driven by `cliq tx apply <id>`.
- bash tool runs under the configured `bashPolicy` and records `BashEffect`s to the active tx.
- `cliq tx validate <id>` and `cliq tx approve <id>` complete the manual workflow.
- Crash recovery converges leftover transactions at runner startup.

## Scope

**In scope:**
1. Runner integration (open at turn start; finalize → validate → approve → apply at turn end).
2. Coordinator API additions: `finalizeTx`, `validateTx`, `approveTx`, `recoverAtStart`.
3. CLI subcommands `cliq tx validate <id>`, `cliq tx approve <id> [--override <name>] [--override-all] [--allow-validator-error <name>]`.
4. Bash policy enforcement at the bash tool's `execute()` site, with `BashEffect` recording.
5. Tests at the unit and integration levels described in §E.

**Out of scope** (future polish PR):
- `cliq tx diff/show/validators/help` subcommands.
- Stage C crash-convergence e2e integration test.
- AB3a.5 race injection test.
- Stage B B3b disk-error explicit test.
- `cliq run --jsonl` subprocess-based end-to-end test.
- Pre-existing `extensions/loader.test.ts` /tmp/workspace flake.

## Non-goals

- Backward-compatibility invariant. The runner integration may require updates to existing v0.7 callsites and tests; we don't bend the API to preserve byte-for-byte v0.7 behavior. Tests get updated along with the code.

## Architectural Decisions

### A.1 — Tx orchestration is a direct parameter on `createRunner`

Codex CLI's `SessionTask` (codex-rs/core/src/tasks/regular.rs) wraps `run_turn` as a coordinator-above-loop layer. We mirror this with an optional `transactions?: TxRunnerOptions` parameter on `createRunner`. When provided, `runTurn` extracts a turn lifecycle that wraps the existing model+tools body.

Rejected alternatives:
- Hooks (`RuntimeHook.beforeTurn` / `afterTurn`): the existing pipeline swallows hook errors via `console.error`; tx lifecycle errors must propagate.
- Wrapper factory `createTxRunner` over an inner `createRunner`: writer injection needs to share state with tool dispatch, which lives on the inner runner — wrapper has to monkey-patch the registry.

### A.2 — `applyPolicy=interactive` + headless = refuse at session start

Codex CLI hardcodes `AskForApproval::Never` for `codex exec` (codex-rs/exec/src/lib.rs:404). It does not promote-mid-run. We follow the same stance:

> When `transactions.applyPolicy === 'interactive'` and `headless === true`, `createRunner` throws at construction with a clear message: `--tx-apply interactive requires a TTY; use --tx-apply manual-only or auto-on-pass for headless runs`.

This intentionally diverges from `bashPolicy=confirm`, which we keep as "promote to deny under headless." `bashPolicy` is per-call (model can adapt to a denied bash); `applyPolicy` is per-run (set once, hard to recover from mid-turn).

### A.3 — `enforceBashPolicy` lives inside the bash tool's `execute()`

The bash tool reads `context.tx` (see A.5) and:
1. Calls `enforceBashPolicy({ policy: tx.bashPolicy, txMode: tx.mode, headless: tx.headless, confirm? })`.
2. On `deny` → returns `{ status: 'error', meta: { code: 'tx-overlay-error', error: message } }`.
3. On `allow` → snapshots cwd mtimes, runs bash, snapshots again, calls `tx.recordBashEffect(eff)` with the diff.

Rejected alternative: dispatch-level enforcement (Codex's `ToolOrchestrator` pattern). bashPolicy is a single-tool concern; localizing avoids piling tx-specific logic onto the runner's tool dispatch.

### A.4 — Coordinator builds the validator registry on demand inside `validateTx`

`coordinator.validateTx(txId)` reads workspace config (cached on the coordinator at construction), calls `buildValidatorRegistry(cfg.transactions.validators)`, materializes the staged view, runs `runValidators`, persists each result via `onResult: writeValidatorResult(root, txId, result)`, cleans up the staged view, and updates `tx.validators` + `tx.blockingFailures`. One method, one pipeline.

Rejected alternatives:
- Runner builds registry at construction: couples runner to validator config.
- Caller passes registry per-call: pushes work onto every callsite (manual `cliq tx validate` would have to build it too).

### A.5 — `ToolContext.tx?: { mode, bashPolicy, txId, headless, recordBashEffect }`

A single optional namespace, presence/absence cleanly signals tx-on/off. Future tx-aware tools (e.g., a search tool that prefers overlay content) read from the same struct without growing `ToolContext` per-feature.

```ts
export type ToolContext = {
  cwd: string;
  session: Session;
  signal?: AbortSignal;
  writer?: WorkspaceWriter; // existing — Phase 3
  tx?: ToolContextTxFacade;
};

export type ToolContextTxFacade = {
  mode: 'edit';
  bashPolicy: TxBashPolicy;
  txId: string;
  headless: boolean;
  recordBashEffect(eff: BashEffect): Promise<void>;
};
```

### A.6 — Recovery runs at runner construction AND at every CLI `tx` subcommand entry

`coordinator.recoverAtStart(ctx)` is idempotent (no-op when nothing to do) and cheap (scan `<txRoot>/tx_*` dirs, reread state.json). Calling it from both places means orphans never accumulate silently regardless of how the user re-enters.

Cross-session guard: tx artifacts carry `tx.sessionId`. `recoverAtStart` filters to `tx.sessionId === ctx.session.id`. Cross-session orphans get a `recovery.json` with `action: 'cross-session-skipped'`; runner emits one runtime warning event per skip so JSONL consumers see them. No state mutation happens across session boundaries in v0.8.

### A.7 — `applyPolicy=manual-only` + `auto: per-turn` is refused at config-load

The combination "auto-open every turn but never auto-apply" produces dangling tx artifacts every turn with no clear lifecycle. We refuse it at workspace config validation time:

> `transactions.applyPolicy=manual-only requires transactions.auto=manual; per-turn auto-open with manual-only apply produces dangling transactions.`

Same error shape if the CLI override is invalid: `--tx-apply manual-only` requires `--tx edit` to be paired with explicit `cliq tx open` (auto: manual at runtime).

### A.8 — On `applyPolicy=interactive` user-no, auto-abort at turn end

When the prompt at turn end returns no, runner calls `coordinator.abortTx({ reason: 'user-abort' })` instead of leaving the tx pending. Symmetric with `auto-on-pass` which closes the tx at turn end either way.

### A.9 — `BashEffect` persistence: append-only JSONL

`<txDir>/bash-effects.json` (JSONL, one effect per line). Helpers `appendBashEffect(root, txId, eff)` and `readBashEffects(root, txId)` mirror `appendAudit`/`readAudit` from store.ts. Cleanest lifecycle: independent of apply-progress, captures effects from staging time forward, no contention with state.json.

## Data flow

### Per-turn implicit tx (auto-on-pass, the most common path)

```
runTurn(session, userInput)
├─ tx-runner.openTurnTx(ctx, opts)
│  ├─ coordinator.openTx({ explicit: false }) → tx, sets activeTxId, NO tx-opened record
│  └─ emit 'tx-staging-start' { txId, trigger: 'auto-turn' }
├─ build OverlayWriter for tx (createOverlayWriter(cwd, <txDir>/overlay))
├─ existing turn body runs:
│  ├─ checkpoint, model loop, tool dispatch
│  └─ each definition.execute(action, { cwd, session, signal, writer, tx: facade })
├─ tx-runner.finishTurnTx(ctx, opts, signal)
│  ├─ coordinator.finalizeTx(txId)
│  │  ├─ computeDiff(cwd, overlayRoot) → write diff.json
│  │  ├─ tx.diffSummary = summarizeDiff(diff)
│  │  ├─ state='finalized'
│  │  └─ emit 'tx-finalized' { diffSummary }
│  ├─ coordinator.validateTx(txId)
│  │  ├─ buildValidatorRegistry(cfg.transactions.validators)
│  │  ├─ materializeStagedView(cwd, overlayRoot, stagedTarget, bindPaths, copyMode)
│  │  ├─ runValidators({ txId, registry, workspaceView: stagedTarget, realCwd: cwd, onResult: persist })
│  │  ├─ cleanupStagedView(stagedTarget)
│  │  ├─ tx.validators = results, tx.blockingFailures = blocking-failures
│  │  ├─ state='validated'
│  │  └─ emit 'tx-validated' { validators, blockingFailures }
│  ├─ if applyPolicy='auto-on-pass' and blockingFailures.length === 0:
│  │  └─ coordinator.approveTx(txId) → state='approved'
│  ├─ if applyPolicy='interactive':
│  │  ├─ const ok = await opts.confirmApply!()
│  │  ├─ if !ok: coordinator.abortTx({ reason: 'user-abort' }) → emit 'tx-aborted'; return
│  │  └─ coordinator.approveTx(txId)
│  ├─ coordinator.applyTx(txId)
│  │  ├─ runApplyTx (Stage A→B→C)
│  │  ├─ session.records ← tx-applied (deterministic id)
│  │  ├─ activeTxId cleared
│  │  └─ emit 'tx-applied' { ... }
│  └─ cleanupOverlay(<txDir>/overlay)
└─ return finalMessage
```

### Branch points at end-of-turn

- **Blocking validator failures, no overrides**: `coordinator.approveTx` returns rejected; runner calls `coordinator.abortTx({ reason: 'validator-fail' })`; emit 'tx-aborted'. No edits land.
- **`applyPolicy='manual-only'` (auto: manual only, per A.7)**: not reachable on auto-per-turn path. On explicit tx, runner doesn't auto-finalize at turn end; user drives `cliq tx validate/approve/apply`.
- **`ApplyConflict` from `applyTx`**: result `{ ok: false, error: 'conflict' }`; runner emits standard 'error' event with code='tx-apply-conflict' (recoverable=true) AND 'tx-aborted' with reason='apply-conflict'. Tx stays 'approved' (Stage A never wrote progress). User can retry `cliq tx apply <id>` after resolving the external change.
- **`ApplyPartial` from `applyTx`**: result `{ ok: false, error: 'partial' }`; runner emits 'error' with code='tx-apply-partial' (recoverable=false). Tx stays 'applied-partial'. The user must run `cliq tx abort --restore-confirmed | --keep-partial`.
- **`coordinator.finalizeTx` or `coordinator.validateTx` throws** (not validator-fail status, but actual exception): runner calls `coordinator.abortTx({ reason: 'staging-error' })`; validators that completed are persisted; emit 'tx-aborted'.
- **Workspace-not-git** (snapshot creation fails inside Stage A): `runApplyTx` throws; caught at `finishTurnTx`; tx stays 'approved'; emit 'error' code='tx-overlay-error'.

### Explicit tx flow

User invoked `cliq tx open <name>` (or workspace config has `auto: manual`).

- The `tx-staging-start` event fires once per tx open, from `coordinator.openTx`. The CLI `tx-open` handler emits it on explicit open; the runner's auto-open path emits it on implicit per-turn open. Subsequent turns reusing an existing explicit tx do NOT re-emit.
- Runner finds existing `activeTxId` at turn start. Calls `coordinator.getActiveTx()` to fetch. If state is `staging`, reuses the existing overlay; runs the turn body; at turn end does NOT finalize/validate/apply (that's user-driven).
- Edits accumulate in the overlay across turns.
- User runs `cliq tx validate <id>`, `cliq tx approve <id>`, `cliq tx apply <id>` manually to drive the lifecycle.

### Recovery at runner startup

`coordinator.recoverAtStart(ctx)`:

```
const orphans = await scanForRecovery(root)
const ownSession = orphans.filter(a => a.tx.sessionId === ctx.session.id)
const crossSession = orphans.filter(a => a.tx.sessionId !== ctx.session.id)
for orphan in ownSession: recoverApply or recoverAbort
for orphan in crossSession: writeRecoveryRecord(root, { txId, action: 'cross-session-skipped', warning: 'tx originated in a different session', ts })
emit one 'error' event per cross-session skip (recoverable=true, message includes txId)
```

Same logic runs at the start of every CLI `tx` subcommand handler.

## Error handling

| Source | Channel | Effect |
|---|---|---|
| `recoverAtStart` failure (IO error reading tx dir) | thrown from `createRunner` constructor | Runner construction fails; CLI exits with `ReportedCliError({ exitCode: 2 })` |
| `coordinator.openTx` failure | thrown from `runTurn` | Turn aborts before model runs |
| `OverlayWriter.replaceText` failure | tool returns `{ status: 'error' }` | Model sees error, can adapt |
| `bashPolicy=deny` rejection | tool returns `{ status: 'error', meta.code: 'tx-overlay-error' }` | Model sees error |
| `coordinator.finalizeTx`/`validateTx` exception | runner catches, calls `abortTx({ reason: 'staging-error' })` | Tx aborted; emit 'tx-aborted' |
| Validator returns `status: 'fail'` (blocking) | normal flow | `approveTx` blocks; runner aborts with reason='validator-fail' unless overrides cover |
| `ApplyConflict` | structured `applyTx` result | Tx stays 'approved'; emit 'error' code='tx-apply-conflict'; emit 'tx-aborted' |
| `ApplyPartial` | structured result | Tx stays 'applied-partial'; emit 'error' code='tx-apply-partial' recoverable=false |
| Cancellation (`signal.aborted`) | existing throwIfCancelled | Cancellation throw; recovery converges next run |

`runHooks` keeps its swallow-via-`console.error` for hook errors (observation only). Tx lifecycle errors propagate.

## CLI subcommand additions

Two new subcommands; same parser pattern as Task 47:

```
cliq tx validate <txId>              [--json] [--headless]
cliq tx approve  <txId>              [--override <name>]... [--override-all] [--allow-validator-error <name>]... [--reason "..."] [--json] [--headless]
```

`tx-validate` ParsedArgs:
```ts
{ cmd: 'tx-validate'; txId: string; json?: boolean; headless?: boolean }
```

`tx-approve` ParsedArgs:
```ts
{
  cmd: 'tx-approve';
  txId: string;
  overrides: string[];
  overrideAll?: boolean;
  allowValidatorError: string[];
  reason?: string;
  json?: boolean;
  headless?: boolean;
}
```

`cliq tx approve` semantics:
- `--override <name>` per-failed-validator override; repeatable.
- `--override-all` overrides every blocking failure (still requires `--reason`).
- `--allow-validator-error <name>` permits validators with `status: 'error'` (not `fail`) to be treated as pass.

Handler invokes `coordinator.approveTx(txId, { overrides, overrideAll, allowValidatorError, reason })`. On success, transitions tx to `approved`. On failure (uncovered blocking failures), returns structured error. JSON output mirrors existing Task 47 patterns.

## Risks and mitigations

- **Risk**: cross-session recovery footgun. **Mitigation**: A.6 + cross-session-skipped recovery.json + warning event; user gets visibility without state corruption.
- **Risk**: `applyPolicy=interactive` UX in TTY mode is hard to get right (diff preview, validator results display). **Mitigation**: v0.8 ships a minimal Y/n prompt with a one-line summary (`Apply tx_xxx? (N files changed, 0 blocking failures): [y/N]`). Richer review can be added in a follow-up.
- **Risk**: model emits a turn that touches no files; runner still opens implicit tx, generates empty diff. **Mitigation**: `coordinator.finalizeTx` detects `diff.files.length === 0` and short-circuits to abort with reason='user-abort'-equivalent (no validators, no apply). `tx-aborted` emitted with a "no edits" marker. Cleaner than producing an empty applied tx.
- **Risk**: bash tool runs concurrently with the OverlayWriter, both touching the same file. **Mitigation**: documented in design that bash modifies cwd directly while OverlayWriter stages in `<txDir>/overlay`. Stage A's preflight catches conflicts (cwd content !== entry.oldContent) and emits ApplyConflict. The BashEffect record makes the source of the conflict visible. v0.8 doesn't pre-validate intersections; this is left for a future "detect bash-overlay collision" enhancement.

## Acceptance criteria

- `cliq --tx edit "..."` against a clean workspace runs end-to-end: opens implicit tx, edits go through overlay, validators run, apply lands changes, all events emitted.
- `cliq tx open foo && cliq "..." && cliq tx apply <id>` works: explicit tx accumulates across turns, manual apply lands.
- `cliq tx validate <id>` and `cliq tx approve <id>` work as CLI entry points to those coordinator methods.
- `cliq run --jsonl --tx edit "..."` emits `tx-staging-start`, `tx-finalized`, `tx-validated`, `tx-applied` (or `tx-aborted`) in the right order.
- Headless run with `--tx-apply interactive` errors out cleanly at construction with the documented message.
- Bash tool under `bashPolicy=passthrough` records `BashEffect` to `<txDir>/bash-effects.json`.
- All new/modified tests green; build clean.
