# v0.9 Control Plane Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Ship `v0.9.0` as one control-plane release: make TX review usable, make approvals payload-aware, add workspace command hooks, connect hooks to approval requests, and finish interactive TX bash confirmation.

**Architecture:** Keep the existing v0.8 TX runtime as the file-change gate. Add a small TX inspection layer for human/JSON review, then introduce an approval request model that the runner can use before tool execution and before TX apply. Command hooks are a separate workspace-configured control plane; they can block, add context, or satisfy approval requests, but core policy and TX validators remain authoritative.

**Tech Stack:** TypeScript, Node.js `node:test`, ESM modules, `child_process.spawn`, existing workspace config parser, existing TX store/coordinator/runtime events. No new runtime dependencies.

## Worktree And Branch Workflow

Use a dedicated implementation worktree, but do not require one worktree per PR.

Default mode is linear execution:

- Reuse one implementation worktree, recommended path: `../cliq-agent-v0.9-impl`.
- After each PR is reviewed and merged, update the worktree to the latest merged base and create the next PR branch in the same worktree.
- Do not delete the worktree unless it is stale, broken, or no longer useful.
- Stale means the worktree's branch base is more than one merged PR behind `origin/main`, or the worktree no longer builds without cleanup such as `git clean -fdx`.

Parallel mode is optional:

- Create multiple worktrees only when two branches will be actively developed or reviewed at the same time.
- Do not parallelize dependent slices by default. PR B depends on PR A's merged shape; PR C depends on PR B's merged approval model; PR D depends on PR C's hook runner; PR E depends on PR C/D hook plumbing.
- If the team intentionally wants stacked PRs, document the stack base in the PR description.

Baseline rule:

1. First land this planning branch as a docs-only baseline PR, or otherwise make sure these docs exist on `main`.
2. Do not start PR A from an unmerged planning branch unless the team intentionally wants stacked PRs.
3. Default workflow is sequential, not stacked: PR B starts after PR A is merged; PR C starts after PR B is merged; continue the same pattern through PR F.

Create the reusable implementation worktree for PR A:

```bash
REPO_ROOT="$(git rev-parse --show-toplevel)"
IMPLEMENTATION_WORKTREE="$REPO_ROOT/../cliq-agent-v0.9-impl"
cd "$REPO_ROOT"
git fetch origin
git worktree add "$IMPLEMENTATION_WORKTREE" -b cliq/v0.9-tx-review-polish origin/main
cd "$IMPLEMENTATION_WORKTREE"
```

After PR A is reviewed and merged, reuse the same worktree for PR B:

```bash
REPO_ROOT="$(git rev-parse --show-toplevel)"
IMPLEMENTATION_WORKTREE="$REPO_ROOT/../cliq-agent-v0.9-impl"
cd "$IMPLEMENTATION_WORKTREE"
git status --short
# Continue only when the worktree is clean.
git fetch origin
git switch --detach origin/main
git switch -c cliq/v0.9-approval-model
```

Repeat the same pattern for PR C through PR F, always creating the next branch from updated `origin/main` after the previous PR is merged.

If a parallel branch is intentionally needed:

```bash

COMMAND_HOOKS_WORKTREE="$REPO_ROOT/../cliq-agent-v0.9-command-hooks"
cd "$REPO_ROOT"
git fetch origin
git worktree add "$COMMAND_HOOKS_WORKTREE" -b cliq/v0.9-command-hooks origin/main
cd "$COMMAND_HOOKS_WORKTREE"
```

Use this only when the branch is genuinely independent or when the team has explicitly accepted a stacked-PR flow.

Per-PR execution rules:

- Keep each implementation branch limited to its PR slice.
- Do not carry unfinished files from the previous worktree into the next one.
- If a later slice needs a change from an earlier unmerged PR, wait for that PR to merge instead of duplicating the change.
- Run the focused tests listed in that PR slice before opening review.
- Run `npm test` and `npm run build` before marking the PR ready.
- Remove completed worktrees only when they are no longer needed, or when keeping them risks stale local state.

Emergency exception:

- If PR A exposes a small shared helper needed by PR B, add it to PR A before review instead of creating a parallel helper in PR B.
- If two slices become tightly coupled during implementation, stop and update this plan before merging them into one PR.

## Target Data Flow

Tool execution after this release:

```text
model action
  -> registry.resolve(action)
  -> build ApprovalSubject(kind="tool")
  -> command hooks: PreToolUse
  -> approvalEngine.decide(subject)
  -> if ask: command hooks: PermissionRequest
  -> if still ask: user confirm callback
  -> if allow: definition.execute(action, context)
  -> command hooks: PostToolUse
  -> in-process extension hooks remain available around the same lifecycle
```

TX apply after this release:

```text
tx state
  -> finalize when state=staging
  -> validate when state=finalized
  -> read TX review snapshot
  -> command hooks: TxApplyReview
  -> approval subject: kind="tx-apply"
  -> rich human prompt when policy asks
  -> approve/apply when allowed
```

Core deny ordering:

```text
core hard deny
  -> cannot be overridden by hooks

hook deny
  -> overrides policy allow

hook allow
  -> can satisfy policy ask
  -> cannot override read-only hard deny or validator blocking failures
```

Create `src/workspace/transactions/inspect.ts` with these exported types and functions:

```ts
export type TxReviewSnapshot = {
  tx: Transaction;
  diff: Diff | null;
  audit: AuditEntry[];
  bashEffects: BashEffect[];
  validatorResults: ValidatorResultSummary[];
  validatorArtifactResults: Array<FullValidatorResult>;
  validatorArtifactErrors: string[];
  artifactRef: string;
};

export async function readTxReviewSnapshot(opts: { root: string; txId: string }): Promise<TxReviewSnapshot>;
export function formatTxDiff(snapshot: TxReviewSnapshot): string;
export function formatTxShow(snapshot: TxReviewSnapshot): string;
export function formatTxValidators(snapshot: TxReviewSnapshot): string;
export function formatTxApplyReview(snapshot: TxReviewSnapshot): string;
```

Implementation rules:

- Read transaction state with `readTxState(root, txId)`.
- Read diff with `readDiff(root, txId)`.
- Read audit with `readAudit(root, txId)`.
- Read bash effects with `readBashEffects(root, txId)`.
- Read validator artifact files from `validatorsDir(root, txId)`.
- Import `ValidatorResult as FullValidatorResult` from `src/validators/types.ts`; it is the existing full artifact result type.
- Populate `validatorResults` from `tx.validators ?? []`; it uses the existing `ValidatorResultSummary` from `src/workspace/transactions/types.ts`.
- Populate `validatorArtifactResults` from validator artifact JSON files when present; it uses the full `ValidatorResult` from `src/validators/types.ts`.
- Sort validator artifact filenames alphabetically before reading.
- Return `validatorArtifactErrors` instead of throwing on invalid validator JSON.
- Throw `new Error("tx not found: <txId>")` when state is missing.

- [ ] **Step 4: Implement deterministic human formatters**

Formatter output requirements:

- `formatTxDiff` prints one block per diff entry: `M path`, `A path`, or `D path`.
- For modify entries, print a compact line count summary and omit full file content.
- `formatTxShow` prints tx id, state, workspace id, session id, createdAt, updatedAt, diff summary, validator summary, bash effect count, and artifact ref.
- `formatTxValidators` prints one line per validator: `PASS blocking tsc 123ms`.
- `formatTxApplyReview` prints the prompt body used by interactive apply and includes blocking failures and bash effects.
